# 前后端边界说明

## 目的

降低后续接手、真实数据替换和系统集成成本，避免同一业务规则散落在数据、配置和渲染代码里。

## 一句话原则

- `dashboard-data.js` 决定“有哪些页面、区域、图表”。
- `dashboard-config.js` 决定“筛选项、图表行为、表格模板、限额和展示规则”。
- `dashboard-domain.js` 决定“业务分类、业务分组、模拟测算基础选项和领域口径”。
- `dashboard-utils.js`、`app.js`、领域渲染模块、`dashboard-simulation.js`、`dashboard-processes.js`、`dashboard-renderers.js`、`dashboard-events.js` 和 `styles.css` 决定“怎么渲染、怎么交互、怎么排版”。

## 当前页面结构

当前一级页面只有三类：

- 利率风险
- 流动性风险
- 业务变动分析

当前数据结构已经压平为三页、三个主 block，不再保留以下历史层级：

- 核心风险指标
- 缺口风险
- 现金流错配

## 数据与配置层职责

以下内容优先放在数据或配置层：

- 页面、区域、图表、表格是否存在
- 图表顺序和归属
- 图表标题、区域标题、表格标题
- 共享筛选项和局部筛选项
- 图表行为：图表类型、表格类型、是否全宽、是否可模拟测算
- 管理限额、图例选项、默认选项
- 业务类型和业务分组

同一 block 下如果多个区域需要合并展示，必须在 `dashboard-data.js` 中配置相同的 `groupKey`，不能再依赖区域中文标题刚好相同来隐式合并。

管理限额必须在 `dashboard-config.js` 中通过 `managementLimits[].widgetSeqs` 绑定图表，不能通过标题关键词匹配。

业务变动分析当前采用利率风险资产负债结构口径，按生息资产、付息负债和表外衍生品组织，对应 `dashboard-config.js` 中 `pageBehavior.业务变动分析.analysisPerspective = "interestBalanceStructure"`。流动性风险口径下的业务流入、流出、到期现金流和资金融入分析，不放进业务变动分析页复用这套表格，而是在流动性风险页面的“未来逐日资金流”“流动性缺口”“资金融入”等图表中表达。

## 渲染层职责

以下内容放在渲染层文件：

- `dashboard-utils.js`：共享工具函数、日期处理、筛选 key、mock 数值包装和签名计算
- `app.js`：页面渲染骨架、通用图表/表格绘制、筛选状态和布局状态
- `dashboard-interest-renderers.js`：利率风险图表和数据视图渲染
- `dashboard-business-renderers.js`：业务变动分析图表、结构表和明细表渲染
- `dashboard-liquidity-renderers.js`：流动性、资金融入和债券投资图表渲染
- `dashboard-simulation.js`：模拟测算弹窗、草稿、结果摘要和图表模拟叠加线
- `dashboard-processes.js`：EVE、LCR/NSFR、重定价缺口率计算过程拆解
- `dashboard-renderers.js`：图表、数据、表格渲染注册，只保留当前配置实际引用的 renderer kind
- `dashboard-events.js`：DOM 事件绑定
- `styles.css`：布局、间距、颜色、响应式

## 明确禁止

不要在 `app.js` 中做这些事：

- 临时新增、删除、重排业务图表
- 运行时隐藏已废弃组件
- 运行时压平旧层级
- 临时修正业务标题
- 临时补业务分类、业务分组或套期项目

这些问题应回到 `dashboard-data.js`、`dashboard-config.js` 或 `dashboard-domain.js` 解决。

## 当前护栏

`scripts/validate_dashboard.py` 已经检查：

- 页面结构和统计数量一致
- 旧层级不能回潮
- 已删除组件序号不能回潮
- `blockDisplay`、`areaDisplay` 不再承载逻辑
- 同名区域必须显式配置 `groupKey`
- 管理限额必须通过 `widgetSeqs` 绑定，不能保留 `matchTitles`
- `dashboard-domain.js` 与业务类型图例保持一致
- `dashboard-domain.js` 必须先于 `app.js` 加载
- `dashboard-utils.js` 必须在 `app.js` 之前加载
- 脚本加载顺序必须保持为 `dashboard-utils.js`、`app.js`、`dashboard-simulation.js`、`dashboard-processes.js`、`dashboard-business-renderers.js`、`dashboard-interest-renderers.js`、`dashboard-liquidity-renderers.js`、`dashboard-renderers.js`、`dashboard-events.js`
- 除 `sourceOnly` 节点外，所有 widget 都必须在 `widgetBehavior` 中声明 `chartKind` 或 `tableKind`
- `app.js` 不能重新承载模拟测算、计算过程弹窗、领域图表渲染或 DOM 事件绑定实现
- `dashboard-renderers.js` 注册的 renderer kind 必须与 `dashboard-config.js` 中实际使用的 kind 完全一致，不允许多注册旧功能

常规检查命令：

```powershell
npm run check
```
