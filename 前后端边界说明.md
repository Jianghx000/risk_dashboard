# 前后端边界说明

## 目的

降低后续接手、真实数据替换和系统集成成本，避免同一业务规则散落在数据、配置和渲染代码里。

## 一句话原则

- `dashboard-data.js` 决定“有哪些页面、区域、图表”。
- `dashboard-config.js` 决定“筛选项、图表行为、表格模板、限额和展示规则”。
- `dashboard-domain.js` 决定“业务分类、业务分组、模拟测算基础选项和领域口径”。
- `dashboard-utils.js`、`app.js`、领域渲染模块、`dashboard-simulation.js`、`dashboard-processes.js`、`dashboard-renderers.js`、`dashboard-events.js` 和 `styles.css` 决定“怎么渲染、怎么交互、怎么排版”。

## 当前页面结构

当前一级页面只有三类：

- 利率风险
- 流动性风险
- 业务变动分析

当前数据结构已经压平为三页、三个主 block，不再保留以下历史层级：

- 核心风险指标
- 缺口风险
- 现金流错配

## 数据与配置层职责

以下内容优先放在数据或配置层：

- 页面、区域、图表、表格是否存在
- 图表顺序和归属
- 图表标题、区域标题、表格标题
- 共享筛选项和局部筛选项
- 图表行为：图表类型、表格类型、是否全宽、是否可模拟测算
- 管理限额、图例选项、默认选项
- 业务类型和业务分组

同一 block 下如果多个区域需要合并展示，必须在 `dashboard-data.js` 中配置相同的 `groupKey`，不能再依赖区域中文标题刚好相同来隐式合并。

管理限额必须在 `dashboard-config.js` 中通过 `managementLimits[].widgetSeqs` 绑定图表，不能通过标题关键词匹配。具体限额通过 `entries[]` 按机构、币种、比较符、数值、单位及可选附加筛选口径维护。顶部机构和币种均必须单选；图中只展示完全匹配当前机构、币种及附加口径的限额线，并通过限额图例注明适用范围。没有条目完全匹配时不展示限额。

业务变动分析保留一个一级入口，在页内通过按钮切换利率风险、流动性风险两个分析视角；每个视角下再按 `dashboard-data.js` 的 area 划分为存量业务、新发生业务、到期业务三个页面分区。默认视角由 `dashboard-config.js` 中 `pageBehavior.业务变动分析.analysisPerspective` 决定，可选视角由 `analysisPerspectiveOptions` 声明；两套项目分类、图表指标、结构表字段和明细字段统一维护在 `dashboard-domain.js` 的 `businessAnalysisPerspectives`，一览表和明细表分别通过 `structureMetricColumnsByScope`、`detailColumnsByScope` 按业务状态配置。利率风险视角按生息资产、付息负债和表外衍生品组织；流动性风险视角按资产、表外收入、负债、表外支出组织 23 个明细业务类别，并与“未来逐日资金流”共用同一分类配置，未来逐日资金流图例按“资产及表外收入”“负债及表外支出”分段。流动性风险一级页面继续承载 LCR、NSFR、流动性缺口和未来逐日资金流等风险指标，不与业务变动分析中的存量、新发生、到期变动分析混用。

新发生业务和到期业务不直接读取真实发生/到期流水，而是通过相邻月末截面明细轧差推导。系统逐月比较本月末与上月末，分别形成每个统计月独立的新发生、到期业务明细表；月度趋势图按统计月份汇总对应月度明细表，一览表和明细清单汇总所选月末区间内各统计月的月度明细表，不按所选区间首尾时点重新轧差，也不再使用连续两日轧差。新发生、到期的月度图、一览表和明细清单均提供“口径说明”，每个组件只展示自身计算逻辑；按钮通过 `widgetBehavior[].methodologyKey` 绑定，说明内容统一维护在 `dashboard-domain.js` 的 `businessChangeMethodology`。

## 渲染层职责

以下内容放在渲染层文件：

- `dashboard-utils.js`：共享工具函数、日期处理、筛选 key、mock 数值包装和签名计算
- `app.js`：页面渲染骨架、通用图表/表格绘制、筛选状态和布局状态
- `dashboard-interest-renderers.js`：利率风险图表和数据视图渲染
- `dashboard-business-renderers.js`：业务变动分析图表、结构表和明细表渲染
- `dashboard-liquidity-renderers.js`：流动性、资金融入和债券投资图表渲染；未来资金流业务选择、穿透明细状态及明细筛选函数也归属本文件，不得放回 `app.js`
- `dashboard-simulation.js`：模拟测算弹窗、草稿、结果摘要和图表模拟叠加线
- `dashboard-processes.js`：EVE、LCR/NSFR、流动性比例和重定价缺口率的计算过程拆解；资产负债重定价久期缺口仅保留风险图表，不支持归因拆解；所有拆解弹窗统一支持比较基期选择，未手动选择时默认以上一期为基期
- `dashboard-renderers.js`：图表、数据、表格渲染注册，只保留当前配置实际引用的 renderer kind
- `dashboard-events.js`：DOM 事件绑定
- `styles.css`：布局、间距、颜色、响应式

## 明确禁止

不要在 `app.js` 中做这些事：

- 临时新增、删除、重排业务图表
- 运行时隐藏已废弃组件
- 运行时压平旧层级
- 临时修正业务标题
- 临时补业务分类、业务分组或模拟测算业务选项

这些问题应回到 `dashboard-data.js`、`dashboard-config.js` 或 `dashboard-domain.js` 解决。

## 当前护栏

`scripts/validate_dashboard.py` 已经检查：

- 页面结构和统计数量一致
- 旧层级不能回潮
- 已删除组件序号不能回潮
- `blockDisplay`、`areaDisplay` 不再承载逻辑
- 同名区域必须显式配置 `groupKey`
- 管理限额必须通过 `widgetSeqs` 绑定并使用 `entries[]` 维护机构、币种和附加口径，不能保留 `matchTitles` 或旧的 `values`
- `dashboard-domain.js` 与业务类型图例保持一致
- `dashboard-domain.js` 必须先于 `app.js` 加载
- `dashboard-utils.js` 必须在 `app.js` 之前加载
- 脚本加载顺序必须保持为 `dashboard-utils.js`、`app.js`、`dashboard-simulation.js`、`dashboard-processes.js`、`dashboard-business-renderers.js`、`dashboard-interest-renderers.js`、`dashboard-liquidity-renderers.js`、`dashboard-renderers.js`、`dashboard-events.js`
- 除 `sourceOnly` 节点外，所有 widget 都必须在 `widgetBehavior` 中声明 `chartKind` 或 `tableKind`
- `app.js` 不能重新承载模拟测算、计算过程弹窗、领域图表渲染或 DOM 事件绑定实现
- `dashboard-renderers.js` 注册的 renderer kind 必须与 `dashboard-config.js` 中实际使用的 kind 完全一致，不允许多注册旧功能
- 模拟测算只允许当前配置的重定价缺口和流动性缺口两个入口，旧通用模式、套期交易模式、上传入口及其状态和配置不能回潮

常规检查命令：

```powershell
npm run check
```
