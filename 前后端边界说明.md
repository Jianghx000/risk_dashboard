# 前后端边界说明

## 目的
- 降低理解成本、交接成本、真实数据替换成本。
- 避免同一业务规则同时散落在 `dashboard-data/config` 和 `app.js` 中，造成“事实源不唯一”。

## 一句话原则
- `dashboard-data.js / dashboard-config.js` 决定“有什么”。
- `app.js / styles.css` 决定“怎么显示”。
- 当前项目只保留一套实际运行文件，不再维护 `*.v20260410.js / *.v20260416.css` 这类运行版副本。

## 数据定义层职责
以下内容必须优先放在：
- `prototype/dashboard-data.js`
- `prototype/dashboard-config.js`

属于数据定义层的内容包括：
- 一级页面、板块、区域、图表/表格的层级结构
- 图表是否存在、顺序如何、归属哪个区域
- 区域是否需要合并，哪些旧区域应直接删除
- 图表标题、表格标题、区域标题
- 共享筛选项、局部筛选项的归属位置
- 筛选项 options、defaultValues、是否多选
- 哪些图与哪些筛选联动，哪些图不联动
- 图表的业务口径说明

## 前端渲染层职责
以下内容放在：
- `prototype/app.js`
- `prototype/styles.css`

属于前端渲染层的内容包括：
- 页面渲染
- 图表绘制
- 图例点击联动
- 图表/数据切换
- 弹窗交互
- 混合时间轴的画法
- 宽图/窄图布局
- 样式、间距、排版、响应式

## 明确禁止
`app.js` 不应再承担以下职责：
- 新增、删除、重排业务图表
- 重命名业务区域、业务标题
- 运行时把两个区域“视觉合并”为一个区域
- 运行时补写 widget 筛选配置
- 运行时修正本应由 `dashboard-data/config` 定义的业务结构

## 修改规则
如果需求属于“业务结构变化”，先改数据定义层：
- 删除 `时间序列 / 时点 / 时间区间`
- 合并第三层区域
- 调整某块保留哪些图
- 调整筛选项属于共享还是局部

如果需求属于“展示方式变化”，再改前端渲染层：
- 图更宽一些
- 标签间隔显示
- AI 按钮位置
- 模拟测算弹窗样式
- 图例样式和布局

## 当前项目中的事实源
当前应以以下文件作为唯一业务结构事实源：
- `prototype/dashboard-data.js`
- `prototype/dashboard-config.js`

当前 `prototype/app.js` 只负责渲染和交互，不应再包含“运行时重写业务结构”的逻辑。

## 修改流程
1. 先改基础副本
- `prototype/dashboard-data.js`
- `prototype/dashboard-config.js`
- `prototype/app.js`
- `prototype/styles.css`

2. 如有缓存风险，再更新 `prototype/index.html` 的 query string

3. 最后关闭旧页签，重新打开实际入口页检查

## 验收标准
满足以下条件才算边界清晰：
- 看 `dashboard-data/config` 就能看懂当前业务结构
- 删掉 `app.js` 中的结构性补丁后，页面仍然正确
- 新接手的人不需要通过读渲染逻辑，反推页面业务结构
- 不再存在“基础副本正确，但运行版还是旧文件”的双文件分叉风险
